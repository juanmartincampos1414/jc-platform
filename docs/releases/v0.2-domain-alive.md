# v0.2 — Domain Alive

**Tag:** `v0.2-domain-alive`
**Fecha:** 2026-06-28
**Sprint:** 1 + 1.5
**Estado:** Producción ✅

---

## Resumen

Hasta v0.1 el sistema era una aplicación que almacenaba información organizada por módulos.

A partir de v0.2 el sistema es un dominio vivo que produce comportamiento.

La diferencia no es cosmética. Cada acción relevante deja una huella trazable: quién actuó, sobre qué Brand, en qué Campaign, con qué agente IA, qué Creative produjo, qué Asset resultó, qué eventos generó. El sistema empieza a observarse a sí mismo.

---

## Arquitectura del dominio

```
Workspace
    ↓
Brand              ← identidad + Memory
    ↓
Campaign           ← unidad operativa
    ↓
Creative           ← intención
    ↓
Asset              ← ejecución
    ↓
Distribution       ← próximo
    ↓
Performance        ← próximo
    ↓
Insights           ← próximo
    ↓
Recommendations    ← próximo
    ↓
Memory Update      ← loop cerrado
```

Arquitectura de comportamiento:

```
User Action
    ↓
Domain Event (events)
    ↓
Activity (activity_logs)
    ↓
Agent Job (agent_jobs)
    ↓
Memory
    ↓
Learning
```

## Qué quedó funcionando

### Events
Cada acción importante del sistema emite un evento de dominio a la tabla `events`.

Eventos activos:
- `brand.created`
- `campaign.created`
- `creative.created`
- `asset.created`
- `asset.approved`
- `asset.rejected`
- `asset.needs_changes`
- `memory.initialized`
- `agent_job.started`
- `agent_job.completed`
- `agent_job.failed`
- `month.generation.started`
- `month.generation.completed`
- `month.generation.failed`
- `document.signed`

### Agent Jobs
Toda llamada a Claude queda registrada con: modelo, prompt_version, input, output, tokens, duración, costo estimado, status y timestamps.

### Brand Memory
Toda Brand tiene al menos un registro `memory_type: 'brand'` desde el momento en que es creada o reconocida por el sistema.

### Creative → Asset
La cadena está completa y sin huecos. Ningún Asset puede quedar con `creative_id = NULL` a partir de Sprint 1.5. El invariante es enforceado en el adapter.

### JClaude Dual Write
`generate-month` escribe simultáneamente en:
- `jclaude_posts` (legacy, backward compat)
- `creatives` (nuevo dominio)
- `assets` vinculados a su Creative

### Legacy Compatibility
Las tablas `jclaude_posts`, `social_posts`, `post_comments` y `activity_logs` permanecen intactas. Las compatibility views `jclaude_posts_v` y `social_posts_v` permiten que código no migrado siga funcionando.

## Qué aprendimos

**Campaign es la unidad operativa, no los módulos.**
Todo el contenido, el aprendizaje y la inteligencia gira alrededor de Campaign. Dashboard, Social Media y JClaude son vistas sobre el mismo dominio.

**Brand no es contexto, es identidad.**
Brand acumula voz, audiencia, restricciones y Memory. Es el activo más valioso del cliente en el sistema.

**Creative y Asset son entidades distintas y esa distinción importa.**
Creative representa intención (qué se quiso decir). Asset representa ejecución (cómo quedó). Un Creative puede generar múltiples Assets. Un Asset sin Creative es un hueco en el aprendizaje.

**JClaude debe vivir detrás del producto, no delante.**
JClaude es el cerebro. No una pantalla. Su output son Creatives y Assets que fluyen hacia el dominio. Sprint 2B lo convierte en Brand Brain.

**Los eventos no son logging. Son comportamiento.**
La tabla `events` no es un log de sistema. Es el stream de comportamiento del dominio. Es la base sobre la que se construirán Insights, Recommendations y Memory.

**La observabilidad IA es arquitectura, no feature.**
Saber qué prompt corrió, con qué modelo, cuánto tardó, qué produjo y cuánto costó no es debugging. Es la base del aprendizaje continuo del sistema.

## Invariantes del dominio

No pueden romperse en ningún sprint futuro.

1. `workspace_id` en cada tabla — límite de seguridad y multi-tenancy absoluto.
2. Todo Asset pertenece a una Campaign a través de un Creative — nunca directamente al Workspace.
3. `creative_id` en Asset es obligatorio — assets sin creative_id violan el dominio.
4. `rejection_reason` es obligatorio al rechazar o pedir cambios — invariante de aprobación.
5. Brand acumula Memory — toda Brand debe tener al menos un registro en `memories`.
6. Events son inmutables — la tabla `events` no tiene UPDATE ni DELETE policies.
7. Agent Jobs registran toda ejecución IA — ninguna llamada a Claude sin `agent_jobs` record.
8. Tablas legacy son de solo lectura conceptualmente — durante Sprint 1 y 2 son fuente de verdad compartida.

## Nota sobre RUN72 Core OS Engine

A partir de v0.2 es posible identificar un motor reutilizable que trasciende a JC AI Agency:

```
RUN72 Marketing OS Engine
    ↓
JC AI Agency (primera aplicación)
```

El motor está compuesto por:
- Domain Model (Brand / Campaign / Creative / Asset)
- Events (comportamiento inmutable)
- Agent Jobs (observabilidad IA)
- Memory (conocimiento persistente)
- AI Layer (Claude)
- Learning Loop (próximo)

JC AI Agency es la primera implementación. Otras agencias o productos podrán construirse sobre el mismo motor sin volver a empezar desde cero.
